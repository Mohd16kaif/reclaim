2.2 Can a client-side full account deletion be done with the anon key?

NO. The Supabase client uses the anon key (line 8):
  const SUPABASE_ANON_KEY = process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY!;

Supabase's admin.deleteUser() REQUIRES the service_role key, which must
be kept server-side (never in a client app). The only way to delete a
Supabase auth user from a client context is via an Edge Function or a
custom RPC that uses security definer (runs with creator's elevated
privileges). Neither exists in this codebase.

A client-side DELETE or DROP operation on tables could remove data rows
via RLS policies, BUT:
- The Supabase Auth user (auth.users) cannot be deleted from the client.
- No RLS policy or RPC for deletion is defined in the code.
- The user row in the public.users table is keyed by device_id (not the
  auth.uid), so there is no FK relationship from the user's data back to
  the auth user that could be leveraged for cascade deletion.

The paywall is triggered via registerPlacement with placement name

- NO deep link or navigation route for the paywall

  "discount_paywall_trigger"

This means: if a user closes or skips (declines) the paywall during
onboarding, there is literally no way to locate/re-trigger the paywall
from within the app after relaunch. This fully explains the "could not
