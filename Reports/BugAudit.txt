#H1.1  Dual streak-start keys: "streakStartDate" vs "@reclaim_current_streak_start"
       Files: utils/profileStorage.ts:81, screens/NotificationsScreen.tsx:103
              vs utils/statsStorage.ts:361, screens/MainDashboardScreen.tsx:521,

       "streakStartDate" (consistent across statsStorage, userProgress, supabase,

#H2.1  Triple concurrent write/read race on streak keys after relapse
       Files: screens/MainDashboardScreen.tsx:1734-1740
       handleStreakReset writes "streakStartDate" and "currentStreak", then
       a 300ms setTimeout triggers refresh() (calls useDashboardData.fetchData()
       which reads streakStartDate and writes longestStreak) and
       refreshWeeklyStats() (reads streakStartDate and writes weeklyStats).
       Meanwhile syncStreakToSupabase() (line 556) also reads the same keys in
       parallel. All three operations overlap with no coordination.
       Trigger: After every relapse, the streak computation in the dashboard
       has a ~300ms window where stale values could be read or written by
       concurrent async operations. Supabase could receive stale streak data.
       Fix: Serialize the chain: await refresh(), await refreshWeeklyStats(),
       then await syncStreakToSupabase(). Remove the setTimeout delay.

#H2.2  PanicActivatedScreen "Continue" navigates without completing session
       Files: screens/PanicActivatedScreen.tsx:272-274
       The "Continue" button calls navigation.replace("PanicCoach", {remainingSeconds})
       WITHOUT calling completePanicSessionWithGrace() first. The original
       @reclaim_panic_end_timestamp remains in storage. When PanicCoach's timer
       expires and calls completePanicSessionWithGrace(), then navigates to
       PanicLock, PanicLock's timer reads the original @reclaim_panic_end_timestamp
       which still exists and overrides the remaining time to the original duration.
       Trigger: Every time a user manually taps "Continue" on PanicActivatedScreen,
       the remaining timer resets to the original full duration upon reaching
       PanicLock. User sees the timer jump back up.
       Fix: Call completePanicSessionWithGrace() before navigating to PanicCoach.

#H2.3  resolvePendingVerdict + closeOpenPanicSession concurrent on @reclaim_panic_sessions
       Files: screens/MainDashboardScreen.tsx:1807-1810, 1839-1842
       The verdict modal fires await Promise.all([resolvePendingVerdict(...),
       stopPanicSession()]). resolvePendingVerdict reads @reclaim_panic_sessions,
       modifies it, writes it back. If closeOpenPanicSession (called from within
       the verdict flow chain) also reads and writes the same key concurrently,
       the second write overwrites the first writer's changes.
       Trigger: In rare timing conditions where both functions run concurrently,
       one session's data (e.g., verdict or endTimestamp) could be lost due
       to the read-modify-write race on the same AsyncStorage key.
       Fix: Serialize panic sessions modifications with a mutex, or combine
       resolvePendingVerdict and closeOpenPanicSession into a single atomic
       function.

#C4.1  "Today" date derived from UTC via toISOString() instead of local time
        Files: utils/statsStorage.ts:145,185
               screens/MainDashboardScreen.tsx:640,862,919
               screens/NotificationsScreen.tsx:95,179
               utils/userProgress.ts:222
               hooks/useDashboardData.ts:152
               App.tsx:452
       Every location that uses new Date().toISOString().split("T")[0] to get
       "today's date" gets the UTC date, not the local date. A user in UTC-5
       between 19:00-23:59 local time gets tomorrow's UTC date. A user in UTC+10
       between 00:00-09:59 gets yesterday's UTC date.
       Trigger: User in UTC-5 checks in at 8 PM local on July 11. Stored date
       is "2026-07-12" (UTC). Next morning at 7 AM local (July 12, 12:00 UTC),
       the UTC "today" is "2026-07-12" — matches stored date. The user is
       blocked from checking in again for up to 19 hours. Same issue affects
       relapse tracking, calendar display, and notification scheduling.
       Fix: Use local-timezone date string instead:
       const getLocalDate = () => { const d = new Date();
         return `${d.getFullYear()}-${String(d.getMonth()+1).padStart(2,'0')}-${String(d.getDate()).padStart(2,'0')}`;
       };

--- HIGH ---

#H4.1  Six divergent streak calculation implementations
        Files: screens/MainDashboardScreen.tsx:392-401 (Math.floor, setHours)
               utils/statsStorage.ts:374-376 (Math.floor, setHours)
               utils/supabase.ts:254-262 (Math.floor, setHours)
               hooks/useDashboardData.ts:60-66 (Math.round, year/month/date ctor)
               utils/profileStorage.ts:18-23 (Math.round, year/month/date ctor)
               screens/NotificationsScreen.tsx:108-109 (Math.floor, raw Date.now())
       Three different algorithms exist:
       - setHours(0,0,0,0) + Math.floor (MainDashboard, statsStorage, supabase)
       - year/month/date constructor + Math.round (useDashboardData, profileStorage)
       - raw Date.now() without midnight normalization (NotificationsScreen)
       These produce different results for the same input. During DST spring-forward
       (23-hour day), Math.floor gives 0 while Math.round gives 1. Without midnight
       normalization, the raw epoch method can give a different result from both.
       Trigger: On DST transition days, different screens show different streak
       lengths for the same data. The NotificationsScreen streak calculation
       (using raw Date.now()) disagrees with the dashboard's setHours-based
       calculation at certain times of day.
       Fix: Create a single shared getStreakDays() utility with consistent
       DST-safe logic using UTC date math or a well-tested date library.

#H4.2  isDateInRange mixes UTC date strings with local Date bounds
        Files: utils/statsStorage.ts:98-105
       isDateInRange receives a UTC-based date string (from #C4.1) but parses
       it with new Date(dateStr + "T00:00:00") which JS interprets as local
       midnight. The start/end bounds from getDateRange use local-time
       setHours(0,0,0,0). A UTC date string for a day that is "tomorrow" in UTC
       but "today" locally will fall outside the local range filter.
       Trigger: Every date range filter (stats screens, weekly stats) can
       misattribute events to the wrong local day for users in non-UTC timezones.
       Fix: Either store all dates as local YYYY-MM-DD and parse as local, or
       store as UTC and always parse via Date.UTC().

#H4.3  Dual streak-start storage keys (#H1.1) causes inconsistent streak display
        Files: As detailed in #H1.1
       profileStorage.ts and NotificationsScreen.tsx read from
       @reclaim_current_streak_start while everyone else reads from
       "streakStartDate". These are different keys with potentially different
       values or null vs real value.
       Trigger: The profile screen and notifications screen display 0-day
       streak while the dashboard shows the correct streak. User sees
       conflicting streak values across the app.
       Fix: Consolidate to a single key, preferably "streakStartDate".

#H4.4  Calendar generator uses mixed UTC/local date strings
        Files: hooks/useDashboardData.ts:68-113 (generateWeekCalendar)
       The calendar builds dates using local-time constructor methods but then
       converts to ISO strings for comparison. isToday uses local toDateString()
       while relapse lookups use UTC date strings. A relapse stored at 1 AM
       local in UTC+10 gets the previous UTC date and appears on the wrong
       calendar square.
       Trigger: Every user in a positive UTC timezone who has a relapse or
       check-in near midnight sees it on the wrong calendar day.
       Fix: Use consistent local-time date strings throughout the calendar.
