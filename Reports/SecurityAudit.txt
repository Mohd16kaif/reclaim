2. SUPABASE ROW LEVEL SECURITY (RLS)
================================================================================

  Critical context: NO SQL migration files or RLS policy definitions exist
  in the repository. RLS policies must be verified in the Supabase dashboard.
  The following analysis is based on client-side code patterns alone.

2.1 How device_id is derived

  All Supabase writes use getOrCreateUserId() (supabase.ts:31-53) which:
  1. Checks AsyncStorage for a cached "@reclaim_user_id"
  2. If absent, calls supabase.auth.signInAnonymously() and caches the result
  3. If that fails, falls back to getDeviceId() (iOS identifierForVendor)

  The returned value is used as "device_id" in all tables (users, streaks,
  stats, settings). All queries filter by .eq("device_id", deviceId).

  RISK: The function accepts no user input for device_id. It always derives
  from the authenticated session. ✅ This is the correct pattern.

  However, there is a separate "deviceId" key read in App.tsx:466:
    AsyncStorage.getItem("deviceId")
  This is a DIFFERENT storage key from the "@reclaim_device_id" used in
  supabase.ts:22. The App.tsx key ("deviceId") is NEVER written to in the
  current codebase — it would be undefined on a fresh install, meaning the
  deviceId passed to identifyUser() would be null and identifyUser would
  be silently skipped (App.tsx:475 check). This means PostHog identification
  may silently fail on fresh installs.

2.2 fetchStreakPercentile — potential RLS bypass

  File: supabase.ts:382-392
  Function: fetchStreakPercentile(currentStreakDays)
  - Queries supabase.from("streaks").select("*", {count: "exact", head: true})
    with NO device_id filter.
  - This reads the TOTAL count of ALL streak records across all users.
  - If RLS is NOT enabled on the "streaks" table, this would return the true
    global count. If RLS IS enabled (scoping to auth.uid()), it would only
    return the count for the authenticated user's own rows, making the
    percentile calculation zero or incorrect.

  SEVERITY: ⚠ MEDIUM — fetchStreakPercentile assumes either (a) RLS is
  disabled on "streaks" or (b) there is a permissive policy allowing reads
  of all rows. If the intent is to show a global percentile, RLS must have
  a specific policy for this query pattern. If RLS is set to auth.uid(),
  this function returns incorrect results. This should be handled via a
  database function (SECURITY DEFINER) or a dedicated API endpoint.

2.3 All write operations use device_id scoping

  Every upsert/insert/update in supabase.ts includes a device_id filter
  derived from getOrCreateUserId(). No function accepts an external device_id
  as a parameter that could allow one user to write another user's data.

  ✅ This is the correct pattern, assuming RLS policies enforce that the
  device_id in the row matches the authenticated user's auth.uid() or a
  linked identity.

2.4 No RLS policy files in repo

  No SQL files exist in the repository. This means RLS policies are managed
  entirely through the Supabase dashboard, which is an operational risk:
  - No version control for security policies
  - No code review for policy changes
  - Harder to audit or reproduce

  SEVERITY: ⚠ MEDIUM — Recommended to export RLS policies as SQL migration
  files and commit them to the repo so they can be reviewed alongside the code.

2.5 Avatar storage

  File: supabase.ts:507-513
    supabase.storage.from("avatars").upload(userId + ".jpg", ...)
  File: supabase.ts:520-531
    supabase.storage.from("avatars").getPublicUrl(userId + ".jpg")

  The avatar bucket uses public URLs, meaning anyone with the URL can read
  the avatar. The URL is generated from userId (auth UID). This is acceptable
  for avatars (they're meant to be public profile pictures), but verify that
  the Supabase storage bucket has appropriate policies — ideally the bucket
  should be public-read but only authenticated-write (matching auth.uid()).

3. AUTHENTICATION / SESSION HANDLING
================================================================================

3.1 Anonymous auth → Apple Sign-In flow

  File: supabase.ts:31-201

  Flow:
  1. getOrCreateUserId() creates anonymous Supabase session and caches auth.uid()
  2. signInWithApple() calls supabase.auth.linkIdentity() to link Apple credential
     to the existing anonymous session
  3. If linking fails because Apple ID was previously linked to a different
     account, falls back to signInWithIdToken() to sign into the old account
  4. The cached "@reclaim_user_id" is updated to the new user ID

  Security concerns:
  - The getDeviceId() fallback (lines 39, 47, 51) when anonymous sign-in fails
    means a device with no network could use a non-authenticated device ID
    fallback. The device_id used in this case would not correspond to any
    real Supabase auth user, which could cause foreign key constraint failures
    or data integrity issues on the server side.
  - No explicit session check before performing actions. All sync functions
    call getOrCreateUserId() which creates a new anonymous user if none
    exists, rather than checking if a session is valid first. This could
    create orphan user rows.
  
  SEVERITY: ⚠ MEDIUM — The anonymous fallback to device ID means operations
  could proceed without a valid server-side session, potentially creating
  orphan data or failing silently.

3.2 No sign-out function

  There is no supabase.auth.signOut() call anywhere in the codebase. The
  only "reset" is PostHog's resetUser() (analytics.ts:55). If a user wants
  to sign out or switch accounts, there's no code path to do so.

  SEVERITY: ⚠ LOW — Acceptable for a recovery app where users are unlikely
  to switch accounts, but should be a documented design decision.

3.3 Auth state change listener

  No supabase.auth.onAuthStateChange() listener was found. The app relies
  entirely on cached AsyncStorage tokens and periodic anonymous sign-in.
  This means sign-in errors, token expiry, or session revocation may not
  be detected or handled gracefully.

  SEVERITY: ⚠ LOW — The app handles most failures via catch blocks and
  console.log fallbacks, so it won't crash, but session expiry may cause
  silent data loss.

  Context for risk assessment:
